# VA Auto-Pilot 与 Ralph 对比

本文说明为什么要做 VA Auto-Pilot、各自适用场景，以及为什么“目标优先、不写实现步骤”通常优于手写细节 Prompt。

## 结论先行

- 当你要的是治理能力、可追溯性、可恢复性与质量闭环，选 **VA Auto-Pilot**。
- 当你要的是低门槛、极速迭代、轻量流程，Ralph 风格会更快。

两者都优秀，但优化目标不同。

## 设计思路差异

### Ralph 风格

- 用简洁任务清单驱动循环
- 每轮 fresh context
- 通过日志与 git 历史保留最小记忆
- 上手非常快，流程负担低

### VA Auto-Pilot

- 用协议驱动管理 Agent 循环
- 机器可读状态源 + 可读看板投影
- 显式的人类控制面（override）
- 强制质量门禁与停机条件
- 追加式运行记忆（可复用 signal）

## VA Auto-Pilot 更适合的场景

- 多人协作、需要流程治理的项目
- 存在安全/架构/合规风险的项目
- 长周期迭代，需要失败恢复能力
- 需要审计轨迹和可复盘决策链的团队

## Ralph 风格更适合的场景

- 个人实验和小功能冲刺
- 低风险仓库、速度优先
- 快速原型验证

## 为什么“目标优先”优于“手写步骤”

手写步骤本质上把模型限制在你当前的局部路径里，而这条路径往往不如最前沿模型的搜索和规划能力。

更好的方式是提供：

- 硬约束
- 验收标准
- 质量门禁

然后让模型自主规划实现路径。

收益：

- 当代码现实与预设不一致时，模型更容易自适应
- 减少提示词维护成本
- 避免人为过早锁定路径
- 更充分利用模型的规划与工具调用能力

## CLI Agent 并发能力优势

当管理 Agent 能调度 CLI Agent 时，并发就变成运行时优化问题。

- 独立轨道可以并发推进
- 质量门禁充当同步屏障
- 多条轨道可以同时前进而不失控

这在大型仓库中特别有效：API、UI、测试、文档可以并发推进，再在统一门禁处收敛。

## 实操原则

不要让 Agent 复读你的实现步骤。

让它在约束内达成最高标准，再用确定性门禁验证结果。

这就是 VA Auto-Pilot 的运行核心。
